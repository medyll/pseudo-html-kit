# Story S10-01 — Input: HTML5 Constraint Validation + Popover Validation Hints

**Epic:** Form Controls Migration  
**Sprint:** Sprint 10  
**Points:** 5  
**Priority:** Must  

---

## User Story

As a **form developer**, I want to use HTML5 constraint validation API instead of custom JS validators, so that the browser handles validation natively with zero additional script.

## Context

HTML5 provides native validation via `required`, `pattern`, `min`, `max`, and constraint validation API. Current `input-pk.html` uses custom JS to validate. This story migrates to native APIs with Popover hints for error messaging.

## Acceptance Criteria

```gherkin
Given input-pk with [required] attribute
When form is submitted without value
Then :invalid pseudo-class applies (no JS)
  And browser-native error message shows
  
Given input-pk with [pattern] or [min]/[max]
When value violates constraint
Then input.validity.valid === false
  And error popover shows via popovertarget
  
Given input-pk with validation error
When user corrects value
Then input:valid applies
  And error popover auto-dismisses
  
Given older browser without Popover API
When validation fails
Then fallback error <span> shows (no JS error)
```

## Technical Notes

- Use `HTMLFormElement.checkValidity()` and `input.validity` object
- Error messages via Popover API (`popover="auto"` with `popovertarget`)
- Fallback: `display:none` error span + JS event listeners (Safari < 17 path)
- CSS: `:invalid` + `:valid` + `:user-invalid` for visual feedback
- Test with happy-dom mock (Popover API not in happy-dom)

## Out of Scope

- Custom validator plugins
- Async validation (email verification, username availability)
- Form-level validation (multi-field dependencies)

## Dependencies

- None (parallel execution safe)

## Implementation Notes

**Date:** 2026-03-06  
**Commit:** 68915fe — "feat(input-pk): HTML5 constraint validation API + Popover hints"

**Files changed:**
- `src/pseudo-assets/components/atoms/input-pk.html` — Complete component rewrite
  - Removed old `[error]` attribute and custom JS validator
  - Added `<div class="input__error-popover" popover="auto">` for native error hints
  - Added `<span class="input__error-fallback">` for browser fallback (Popover API unavailable)
  - Updated props doc: added `pattern`, `min`, `max` attributes
  - Updated script: native constraint validation with `checkValidity()` + `reportValidity()` exposure
  - CSS: Added `:invalid`, `:user-invalid`, `:valid` selectors + popover styling

- `tests/s10-01-input-validation.client.test.js` — NEW
  - 15 tests covering constraint validation API
  - Tests: required, type mismatch, min/max ranges, checkValidity/reportValidity methods
  - Tests: validation message API, form submission integration, a11y (aria-describedby, aria-label)
  - All tests pass ✅

**Notable decisions:**
1. **Native-first approach:** Removed custom JS validation entirely — browser handles it natively
2. **Popover + Fallback:** Dual-path support for error hints (Popover for modern browsers, span + CSS for older)
3. **Script refactoring:** Minimal JS (event listeners + feature detection) instead of custom validator
4. **CSS layers:** Placed `:invalid` / `:valid` styling in main @scope block for consistent priority
5. **A11y:** `aria-describedby` links input to error element, `role="alert"` + `aria-live="polite"` for screen readers

**Known limitations:**
- Popover API not implemented in happy-dom → tests mock the feature detection
- Validation messages are browser-provided (localization varies)
- No custom validators (by design — fits constraint API philosophy)
- No async validation (username availability checks) — deferred to future story

**Browser support verified:**
- Chrome 118+: Full native Popover API support ✓
- Firefox 128+: Full native Popover API support ✓
- Safari 17.4+: Full native Popover API support ✓
- Older browsers: Fallback span + CSS visuals ✓
