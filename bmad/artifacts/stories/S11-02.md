# Story S11-02 — Textarea: Anchor Positioning for Auto-Resize Hints

**Epic:** Form Controls Migration  
**Sprint:** Sprint 11  
**Points:** 3  
**Priority:** Should  

---

## User Story

As a **form developer**, I want `textarea-pk` to display an auto-resize hint anchored to the bottom-right corner using CSS Anchor Positioning, so that the component communicates its resizability without JS height calculation.

## Context

CSS Anchor Positioning (Chrome 125+) allows elements to be positioned relative to an anchor element declared with `anchor-name`. This story adds an anchored character-count / resize-hint element to `textarea-pk.html` and explores using field-sizing for auto-resize. Falls back gracefully to a static hint.

## Acceptance Criteria

```gherkin
Given textarea-pk with maxlength attribute
When rendered on Chrome 125+
Then .textarea__hint appears anchored below-right of the textarea
  And hint shows current/max character count

Given textarea-pk on any browser
When field-sizing: content is supported
Then textarea auto-grows with content (no JS)

Given textarea-pk on older browser
When anchor positioning is not supported
Then hint shows as static element below the textarea (no JS error)

Given textarea-pk with error attribute
When rendered
Then .textarea__error[role="alert"] is visible (existing behavior preserved)
```

## Technical Notes

- `anchor-name: --textarea-anchor` on `.textarea__field`
- `.textarea__hint { position: absolute; position-anchor: --textarea-anchor; ... }`
- `@supports (anchor-name: --a)` gate
- `field-sizing: content` for auto-grow (progressive enhancement)
- Character count JS: minimal `oninput` handler updating `data-char-count` attribute
- CSS renders count via `content: attr(data-char-count) " / " attr(maxlength)`
- Fallback: static `.textarea__hint` below field (positioned normally)

## Out of Scope

- JS-based auto-resize (height calculation)
- Custom resize handles
- Rich text / markdown rendering
